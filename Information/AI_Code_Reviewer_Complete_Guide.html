📌 Use Case

What Problem Does This Solve?

Code review is the biggest bottleneck in software development. This project eliminates that bottleneck with instant, automated AI reviews on every Pull Request.

⏳

Problem: Slow Reviews

Human code reviews can take hours or days, blocking releases and frustrating developers waiting for feedback on their code.

🎯

Problem: Missed Issues

Reviewers get fatigued. Security vulnerabilities, OWASP risks, and architectural debt slip through inconsistent human reviews.

🔄

Problem: Context Switching

Developers must stop their own coding to review someone else's work — breaking deep focus and reducing overall productivity.

⚡

Solution: Instant Feedback

Bot reviews every PR in ~30 seconds. Developers get detailed, actionable feedback before a human even opens the PR.

🛡️

Solution: Complete Coverage

4 specialized agents run in parallel covering security, static analysis, style, and architecture — nothing is missed.

🧠

Solution: Gets Smarter

After every merged PR, the Learner service captures patterns. Future reviews are personalized to each repo's history.

✨ Features

Production-Grade Capabilities

Built with enterprise-level reliability, security, and scalability from day one.

🚀

Zero Downtime Deployments

Kubernetes rolling updates ensure new pods are healthy before old ones stop. No webhook events are ever missed during code deployments.

📈

Auto Scaling (HPA)

Kubernetes Horizontal Pod Autoscaler scales the orchestrator from 2 to 10 pods automatically during PR traffic spikes.

⚡

Async Everything

All DB and HTTP calls use non-blocking asyncio + asyncpg. Services stay fast under high concurrency with no thread blocking.

🔒

HMAC Signature Verification

Every single webhook is verified using SHA-256 HMAC before processing. Fake or malicious spoofed GitHub events are rejected at the gateway.

🧬

Deduplication Logic

Webhook service deduplicates by repo + PR number + commit SHA preventing duplicate AI analysis calls when GitHub resends events.

🔑

Passwordless AWS Auth (OIDC)

GitHub OIDC integration generates temporary AWS credentials per pipeline run. Zero long-lived secrets stored anywhere in the system.

📊

Full Observability Stack

Prometheus scrapes metrics every 60 seconds. Grafana displays dashboards. LangFuse traces every GPT call with cost, latency, and token counts.

📅

Weekly AI Quality Checks (RAGAS)

Every Monday, RAGAS evaluates comment faithfulness and relevancy automatically. Alerts if AI quality drifts below threshold.

🧠

Self-Improving Reviews

Learner service stores patterns from merged PRs. Style agent context grows over time — reviews get increasingly relevant to each specific repository.

🛡️

Fault Tolerance (2 Replicas)

Every service runs as 2 Kubernetes pods by default. Pod crash = instant traffic failover. No manual intervention required.

🔄 Workflow

End-to-End Workflow

From the moment a developer opens a Pull Request to actionable AI review comments appearing — every step explained.

📤 Inbound: PR Opens

Developer Opens Pull Request

Developer pushes code changes to a feature branch and opens a PR against main. This is the only action required from the developer.

0ms

GitHub Fires Webhook Event

GitHub automatically sends an HTTPS POST request with a JSON payload to your system's public URL (ALB). This is GitHub's built-in webhook mechanism.

~2ms

Gateway Verifies Authenticity

Gateway checks the X-Hub-Signature-256 header using HMAC-SHA256. If the signature doesn't match your secret, request is rejected with 401. Prevents fake/spoofed PR events.

~5ms

Webhook Service Processes Event

Parses the payload, extracts PR number, repo name, commit SHA. Checks if this exact commit was already analyzed. Saves PR record to PostgreSQL. Queues a Celery job in Redis.

~20ms

Instant 202 Returned to GitHub

GitHub receives 202 Accepted immediately. GitHub's 10-second timeout is never hit. Everything from here is background processing.

~25ms

📥 Background: AI Analysis

Celery Worker Picks Up Job

A background Celery worker running independently picks up the queued analysis job from Redis and makes an HTTP call to the Orchestrator service.

~1s

Orchestrator Fetches Code Diff

Makes an authenticated call to GitHub API using a short-lived JWT installation token to fetch the exact +/- line changes in the PR diff.

~2s

4 AI Agents Run in Parallel

LangGraph launches all 4 agents simultaneously — Static Analysis, Security, Style, and Architecture. Each sends the diff to GPT-4o-mini with a specialized system prompt.

~3s start

Findings Merged & Saved

All 4 agent responses (structured JSON arrays) are combined, duplicates removed, enriched with severity levels, and written to the findings table in PostgreSQL.

~26s

Reviewer Posts to GitHub PR

Reviewer service generates GitHub API calls to post inline comments on the exact lines of code, plus a full review summary. Developer is notified by GitHub.

~30s ✅

🔀 After Merge: Learning Loop

📚 How the System Gets Smarter Over Time

When a developer merges the PR, GitHub fires a closed + merged webhook. A separate Celery Learner Worker picks this up and calls the Learner service. It reads all warning and error findings from this PR and upserts them into the patterns table, incrementing frequency counts for recurring issues. The next time the same repo gets a PR, the Style agent's prompt automatically includes these historical patterns as context — making the review aware of what mistakes this specific team tends to make.

🤖 AI Agents

How the 4 AI Agents Work

Each agent receives the same code diff but uses a specialized system prompt. They run simultaneously via LangGraph and each returns a structured JSON array of findings.

🔍 Static Analysis Agent

Prompt: "You are a static analysis tool. Analyze this code diff for..."

  • High cyclomatic complexity functions
  • Unused variables and imports
  • Poor variable / function naming
  • Dead code blocks
  • Magic numbers and hardcoded values

🛡️ Security Agent

Prompt: "You are a security scanner. Find OWASP Top 10 vulnerabilities..."

  • SQL injection vulnerabilities
  • Hardcoded secrets & API keys
  • Insecure deserialization
  • XSS and injection flaws
  • Broken access control patterns

🎨 Style Agent

Special: Prompt includes historical patterns from this repo

  • Code formatting and indentation
  • Readability and clarity issues
  • Naming consistency
  • Repo-specific past patterns (from Learner)
  • Documentation and comment quality

🏗️ Architecture Agent

Prompt: "Find separation of concerns violations, missing error handling..."

  • Separation of concerns violations
  • Missing error handling
  • Tight coupling between modules
  • Single responsibility principle breaches
  • Dependency direction issues

🛠️ Tech Stack

Tools & Technologies

Every tool explained simply — what it is, what it does in this project, and why it was chosen.

⚙️ Application Layer

Python 3.11
Language
The primary programming language for all 5 backend microservices. Python's async/await capabilities make it perfect for I/O-heavy services that call external APIs and databases without blocking.
→ All 5 services written in Python
FastAPI
Web Framework
A modern, high-performance web framework for Python. Handles HTTP routing natively with async support. Much faster than Flask for concurrent requests. Auto-generates API docs.
→ HTTP server for all 5 microservices
Celery
Task Queue
A distributed task queue. Allows the webhook service to queue heavy AI analysis jobs and return a response to GitHub immediately, without waiting 30 seconds. Workers run independently.
→ Background task processing for analysis jobs
LangGraph
AI Orchestration
A framework for building multi-agent AI workflows. Runs all 4 specialized AI agents (Static, Security, Style, Architecture) in parallel and collects their results into a unified output.
→ Manages parallel AI agent execution
OpenAI GPT-4o-mini
AI Model
The large language model that actually reads the code diffs and finds issues. GPT-4o-mini is used for its balance of speed, cost, and capability — ideal for high-volume code review tasks.
→ The core AI brain for all 4 review agents
SQLAlchemy (async)
ORM / Database
A Python database toolkit that provides an object-relational mapper. Uses asyncpg under the hood for non-blocking PostgreSQL queries, preventing database calls from slowing down the services.
→ Reads/writes to PostgreSQL without blocking
httpx
HTTP Client
A modern async HTTP client for Python. Used for all outbound HTTP calls — fetching code diffs from GitHub API, calling internal microservices, and any external requests.
→ All inter-service and GitHub API calls
PyJWT
Authentication
A library that creates signed JSON Web Tokens. The GitHub App authentication requires a JWT signed with the app's private key, valid for 10 minutes, to generate an installation access token.
→ GitHub App authentication tokens

☁️ Infrastructure Layer (AWS)

AWS EKS
Container Orchestration
Amazon Elastic Kubernetes Service. Manages all 5 services as Docker containers across 2 t3.medium nodes. Handles load balancing, pod restarts, rolling deployments, and auto-scaling automatically.
→ Runs all 5 containerized microservices in production
AWS RDS (PostgreSQL 15)
Database
Amazon's managed relational database service. Stores all persistent data: pull request records, AI findings, and learned code patterns. Lives inside a private VPC subnet — unreachable from the internet.
→ Persistent storage for PRs, findings, patterns
AWS ElastiCache (Redis)
In-Memory Queue
Amazon's managed Redis service. Acts as the message broker for Celery. Webhook service pushes jobs in; Celery workers pull them out. Two separate queues: webhook tasks and learning tasks.
→ Celery task queue broker between services
AWS ECR
Container Registry
Elastic Container Registry. A private Docker image repository. GitHub Actions builds images and pushes them here; EKS pulls from here during deployment. Each service has its own ECR repository.
→ Stores Docker images for all 5 services
AWS ALB
Load Balancer
Application Load Balancer. The only publicly accessible entry point. Receives HTTPS webhook events from GitHub and routes traffic into the EKS cluster. Provides the public IP address for the GitHub App webhook URL.
→ Public entry point, routes GitHub webhooks
Terraform
Infrastructure as Code
Declares the entire AWS infrastructure as code. Run one command and it creates the VPC, EKS cluster, RDS, ElastiCache, ECR repos, S3, and all networking. Run another command to destroy everything cleanly.
→ Creates and manages all AWS resources

📊 Observability & Quality Layer

Prometheus
Metrics Collection
An open-source time-series monitoring system. Scrapes /metrics endpoints from all 5 FastAPI services every 60 seconds. Stores request counts, latencies, and error rates over time.
→ Collects HTTP metrics from all services every 60s
Grafana
Dashboards
A visualization tool connected to Prometheus. Creates real-time dashboards showing request rate per service, p99 latency, error rate, and pod health. Accessible via browser after port-forward.
→ Visual dashboards for all service metrics
LangFuse
LLM Tracing
A dedicated observability platform for LLM applications. Traces every single OpenAI API call — the exact prompt sent, the response received, token count, cost in dollars, and latency. Essential for LLM debugging.
→ Traces every GPT-4o-mini call with full context
RAGAS
AI Quality Evaluation
A framework for evaluating RAG systems and LLM outputs. Runs automatically every Monday via a GitHub Actions schedule. Measures faithfulness (does the comment match the code?) and relevancy of AI reviews.
→ Weekly automated AI comment quality scoring

🚀 CI/CD Layer

GitHub Actions
CI/CD Pipelines
5 independent workflow pipelines — one per service. Each triggers only when that service's code changes. Each pipeline has 3 jobs: test → build+push to ECR → deploy to EKS via kubectl.
→ Automated test, build, and deploy on every push
Docker
Containerization
Each service is packaged into a Docker container with all its dependencies. The exact same image runs locally and in production on EKS, eliminating "works on my machine" problems.
→ Containerizes all 5 services for consistent deployment
GitHub OIDC
Passwordless Auth
OpenID Connect integration between GitHub Actions and AWS. Instead of storing long-lived AWS access keys as secrets, OIDC generates temporary credentials dynamically for each pipeline run. Zero stored AWS keys.
→ Secure, keyless AWS authentication for CI pipelines
pytest
Testing
The standard Python testing framework. Each service has its own test suite that runs in the first job of the CI pipeline. Broken code is caught here and never reaches the build or deploy stages.
→ Automated unit/integration tests per service

🔐 Security Design

Perimeter Security

HMAC-SHA256 verification on every single webhook. Requests without valid signatures from GitHub's secret key are rejected at the gateway.

AWS Access (OIDC)

GitHub Actions uses OpenID Connect — generates temporary AWS credentials for each pipeline run. Absolutely zero long-lived access/secret keys stored anywhere.

GitHub App Auth

Short-lived JWT tokens (10-minute expiry) generated per API call using RSA private key. Never stored in database, never reused.

Network Isolation

PostgreSQL and Redis live inside a private VPC subnet. No internet gateway — only the EKS cluster can reach them. Container images in private ECR.
